CQRS

CQRS и Event Sourcing

Также известен как Command Query Responsibility Segregation
Сложность ★★★★☆
Применимость ★★★☆☆

Назначение

CQRS разделяет операции чтения и записи данных на разные модели. Event Sourcing хранит не текущее состояние, а последовательность событий, которые к нему привели. Вместе они образуют мощную архитектуру для сложных доменов.

Проблема

В традиционных CRUD-приложениях одна модель используется и для чтения, и для записи. Это приводит к компромиссам: сложным запросам, проблемам с производительностью и ограниченной аудируемости. При этом часто теряется информация о том, какие действия привели к текущему состоянию.

Решение

CQRS: Команды (Command) изменяют состояние, запросы (Query) читают данные. У каждой операции — своя модель. Команды интенциональны (выражают намерение), запросы возвращают данные без побочных эффектов.

Event Sourcing: Вместо хранения текущего состояния хранится поток событий. Текущее состояние восстанавливается воспроизведением (replay) событий. Это даёт полный аудит и возможность «путешествия во времени».

Пример на Dart

Команды и запросы:

// ---- Commands (изменение состояния) ----
abstract class Command {}

class DepositMoney implements Command {
  final String accountId;
  final double amount;
  DepositMoney(this.accountId, this.amount);
}

class WithdrawMoney implements Command {
  final String accountId;
  final double amount;
  WithdrawMoney(this.accountId, this.amount);
}

// ---- Events (факты) ----
abstract class Event {}

class MoneyDeposited implements Event {
  final String accountId;
  final double amount;
  final DateTime timestamp;
  MoneyDeposited(this.accountId, this.amount, this.timestamp);
}

class MoneyWithdrawn implements Event {
  final String accountId;
  final double amount;
  final DateTime timestamp;
  MoneyWithdrawn(this.accountId, this.amount, this.timestamp);
}

// ---- Event Store ----
class EventStore {
  final List<Event> _events = [];

  void save(Event event) => _events.add(event);

  List<Event> getEvents(String accountId) =>
      _events.where((e) =>
        (e is MoneyDeposited && e.accountId == accountId) ||
        (e is MoneyWithdrawn && e.accountId == accountId)
      ).toList();
}

// ---- Read Model (Query) ----
class AccountBalanceQuery {
  final EventStore store;

  AccountBalanceQuery(this.store);

  double getBalance(String accountId) {
    double balance = 0;
    for (final event in store.getEvents(accountId)) {
      if (event is MoneyDeposited) balance += event.amount;
      if (event is MoneyWithdrawn) balance -= event.amount;
    }
    return balance;
  }
}

// ---- Command Handler ----
class AccountCommandHandler {
  final EventStore store;

  AccountCommandHandler(this.store);

  void handle(Command command) {
    if (command is DepositMoney) {
      store.save(MoneyDeposited(
        command.accountId, command.amount, DateTime.now()));
    } else if (command is WithdrawMoney) {
      store.save(MoneyWithdrawn(
        command.accountId, command.amount, DateTime.now()));
    }
  }
}

void main() {
  final store = EventStore();
  final handler = AccountCommandHandler(store);
  final query = AccountBalanceQuery(store);

  handler.handle(DepositMoney('acc1', 1000));
  handler.handle(WithdrawMoney('acc1', 200));

  print('Баланс: ${query.getBalance('acc1')}'); // 800.0
}

Плюсы

  • Полный аудит всех изменений
  • Модели чтения могут быть оптимизированы независимо
  • Возможность восстановить состояние на любой момент
  • Хорошая масштабируемость (CQRS)

Минусы

  • Значительно увеличивает сложность системы
  • Требует обработки событийной согласованности (eventual consistency)
  • Избыточен для простых CRUD-приложений
  • Сложность управления версиями событий

Связанные принципы

  • Domain-Driven Design — CQRS и Event Sourcing часто применяются в DDD
  • Observer — событийная модель лежит в основе Event Sourcing
