DDD

Domain-Driven Design (Предметно-ориентированное проектирование)

Также известен как DDD
Автор Eric Evans
Применимость ★★★☆☆

Назначение

DDD — это подход к проектированию сложного программного обеспечения, в котором основной фокус направлен на предметную область (domain) и её логику. Код строится вокруг модели предметной области, а не вокруг технических деталей.

Проблема

Сложные бизнес-приложения часто страдают от разрыва между моделью в голове экспертов и реализацией в коде. Разработчики и бизнес-эксперты говорят на разных языках. Технические детали (БД, UI) смешиваются с бизнес-логикой, делая её непонятной и хрупкой.

Решение

DDD предлагает набор стратегических и тактических паттернов:

  • Ubiquitous Language (Единый язык) — единая терминология между разработчиками и экспертами
  • Bounded Context (Ограниченный контекст) — чёткие границы каждой модели
  • Entity (Сущность) — объект с уникальным идентификатором и жизненным циклом
  • Value Object (Объект-значение) — неизменяемый объект, определяемый своими атрибутами
  • Aggregate (Агрегат) — группа объектов с единой точкой входа (Aggregate Root)
  • Repository (Репозиторий) — абстракция хранилища для агрегатов
  • Domain Service (Доменный сервис) — логика, не вписывающаяся в сущности
  • Domain Event (Доменное событие) — значимое событие в домене

Пример на Dart

Модель заказа в интернет-магазине с использованием DDD:

// ---- Value Object ----
class Money {
  final double amount;
  final String currency;

  const Money(this.amount, this.currency);

  Money operator +(Money other) {
    if (currency != other.currency) {
      throw ArgumentError('Разные валюты');
    }
    return Money(amount + other.amount, currency);
  }

  @override
  bool operator ==(Object other) =>
    other is Money && amount == other.amount && currency == other.currency;

  @override
  int get hashCode => Object.hash(amount, currency);
}

// ---- Entity ----
class Product {
  final String id;
  final String name;
  final Money price;

  Product(this.id, this.name, this.price);

  @override
  bool operator ==(Object other) =>
    other is Product && id == other.id;

  @override
  int get hashCode => id.hashCode;
}

// ---- Value Object ----
class OrderLine {
  final Product product;
  final int quantity;
  final Money total;

  OrderLine(this.product, this.quantity)
    : total = Money(product.price.amount * quantity, product.price.currency);
}

// ---- Aggregate Root ----
class Order {
  final String id;
  final List<OrderLine> _lines = [];
  OrderStatus _status;
  final DateTime createdAt;

  Order(this.id, {DateTime? createdAt})
    : _status = OrderStatus.draft,
      createdAt = createdAt ?? DateTime.now();

  OrderStatus get status => _status;
  List<OrderLine> get lines => List.unmodifiable(_lines);

  Money get total => _lines.fold(
    Money(0, 'USD'),
    (sum, line) => sum + line.total,
  );

  void addProduct(Product product, int quantity) {
    if (_status != OrderStatus.draft) {
      throw StateError('Нельзя изменить подтверждённый заказ');
    }
    _lines.add(OrderLine(product, quantity));
  }

  void confirm() {
    if (_lines.isEmpty) throw StateError('Пустой заказ нельзя подтвердить');
    _status = OrderStatus.confirmed;
  }

  void cancel() {
    _status = OrderStatus.cancelled;
  }
}

enum OrderStatus { draft, confirmed, shipped, delivered, cancelled }

// ---- Domain Event ----
class OrderConfirmed {
  final String orderId;
  final DateTime confirmedAt;
  OrderConfirmed(this.orderId, this.confirmedAt);
}

// ---- Repository (interface) ----
abstract class OrderRepository {
  Future<Order?> findById(String id);
  Future<void> save(Order order);
}

// ---- Domain Service ----
class OrderService {
  final OrderRepository repository;

  OrderService(this.repository);

  Future<void> confirmOrder(String orderId) async {
    final order = await repository.findById(orderId);
    if (order == null) throw ArgumentError('Заказ не найден');

    order.confirm();
    await repository.save(order);
    // Публикация доменного события:
    print('Событие: OrderConfirmed(${order.id}, ${DateTime.now()})');
  }
}

Плюсы

  • Код отражает реальную бизнес-логику
  • Единый язык между разработчиками и экспертами
  • Чёткие границы контекстов и агрегатов
  • Код проще поддерживать при сложной бизнес-логике

Минусы

  • Избыточен для простых CRUD-приложений
  • Требует тесного взаимодействия с экспертами домена
  • Высокий порог входа для команды
  • Сложность внедрения в существующий проект

Связанные принципы

  • Clean Architecture — DDD и Clean Architecture отлично дополняют друг друга
  • CQRS / Event Sourcing — часто применяются вместе с DDD
  • Dependency Injection — необходим для внедрения репозиториев и сервисов
