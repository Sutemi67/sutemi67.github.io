Принципы

YAGNI — You Ain't Gonna Need It (Вам это не понадобится)

Происхождение Extreme Programming (XP)
Сложность ★☆☆☆☆
Применимость ★★★★★

Назначение

Не добавляйте функциональность, пока она не понадобится. Создавайте только то, что требуется прямо сейчас. Большинство предположений о будущем не оправдываются, а лишний код становится обузой, которую нужно поддерживать, тестировать и понимать.

Проблема

Желание заложить архитектуру «на вырост» приводит к тому, что код обрастает абстракциями, интерфейсами и конфигурациями, которые никогда не используются. Каждая такая абстракция увеличивает сложность системы и когнитивную нагрузку на разработчиков.

Решение

Писать код, который решает текущую задачу, и только её. Если позже появится новая потребность — вы добавите функциональность в тот момент, когда она действительно понадобится, имея более полное понимание требований. Рефакторинг существующего кода обойдётся дешевле, чем поддержка невостребованных абстракций.

Пример на Dart

Плохо — преждевременная абстракция «на всякий случай»:

abstract class LoggerInterface {
  void log(String message);
}

class ConsoleLogger implements LoggerInterface {
  @override
  void log(String message) => print(message);
}

// Абстракция не нужна, пока файловый логгер не потребовался

Хорошо — только то, что нужно сейчас:

void log(String message) => print(message);

Плюсы

  • Меньше кода — меньше багов
  • Быстрее время разработки
  • Проще поддержка и рефакторинг
  • Низкая когнитивная нагрузка

Минусы

  • Строгий YAGNI может потребовать более крупного рефакторинга позже
  • В некоторых случаях предварительное проектирование дешевле переписывания
  • Требует дисциплины и умения прогнозировать, где абстракция оправдана

Связанные принципы

  • KISS — YAGNI дополняет KISS: не усложняй и не делай лишнего
  • DRY — баланс между YAGNI и DRY: абстрагируйся только когда появилось реальное дублирование
  • Clean Architecture — YAGNI помогает не перепроектировать архитектуру заранее
