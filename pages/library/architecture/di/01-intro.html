DI

Dependency Injection (Внедрение зависимостей)

Также известен как DI, IoC (Inversion of Control)
Сложность ★★☆☆☆
Применимость ★★★★★

Назначение

Объекты не создают свои зависимости самостоятельно — они получают их извне. Это позволяет сделать код слабо связанным, тестируемым и гибким. DI — это реализация принципа инверсии зависимостей (DIP).

Проблема

Когда класс сам создаёт свои зависимости (через new), он жёстко привязывается к конкретным реализациям. Это делает невозможным: подмену реализации для тестов, переиспользование класса в другом контексте, изменение поведения без изменения кода класса.

Решение

Зависимости «внедряются» в класс через конструктор, сеттер или интерфейс. Класс не знает о конкретной реализации — он работает через абстракцию. Сборкой зависимостей занимается специальный компонент: DI-контейнер или композиционный корень (composition root).

Три способа внедрения:

  • Constructor Injection — зависимости передаются через конструктор (самый распространённый)
  • Setter Injection — зависимости устанавливаются через сеттеры
  • Interface Injection — зависимости передаются через метод интерфейса

Пример на Dart

Без DI — класс сам создаёт зависимости:

class OrderService {
  final _logger = ConsoleLogger(); // жёсткая связь
  final _repository = PostgresOrderRepository(); // жёсткая связь

  void process(Order order) {
    _logger.log('Обработка заказа ${order.id}');
    _repository.save(order);
  }
}

// При тестировании нельзя подменить логгер или репозиторий!
// При смене БД нужно менять код OrderService

С DI — зависимости внедряются через конструктор:

abstract class Logger {
  void log(String message);
}

abstract class OrderRepository {
  void save(Order order);
}

class ConsoleLogger implements Logger {
  @override
  void log(String message) => print(message);
}

class PostgresOrderRepository implements OrderRepository {
  @override
  void save(Order order) => print('Сохранение заказа ${order.id}');
}

class OrderService {
  final Logger logger;
  final OrderRepository repository;

  OrderService(this.logger, this.repository);

  void process(Order order) {
    logger.log('Обработка заказа ${order.id}');
    repository.save(order);
  }
}

// ---- Composition Root ----
void main() {
  final logger = ConsoleLogger();
  final repository = PostgresOrderRepository();
  final service = OrderService(logger, repository);

  service.process(Order(1, 'Заказ 1'));

  // Для тестов легко подменить:
  // final testService = OrderService(MockLogger(), MockRepository());
}

Простой DI-контейнер:

class Container {
  final _factories = <Type, Function>{};

  void register<T>(Function factory) {
    _factories[T] = factory;
  }

  T resolve<T>() {
    final factory = _factories[T];
    if (factory == null) throw ArgumentError('${T} не зарегистрирован');
    return factory() as T;
  }
}

void setupContainer(Container container) {
  container.register<Logger>(() => ConsoleLogger());
  container.register<OrderRepository>(() => PostgresOrderRepository());
  container.register<OrderService>(() =>
    OrderService(
      container.resolve<Logger>(),
      container.resolve<OrderRepository>(),
    ));
}

void main() {
  final container = Container();
  setupContainer(container);
  final service = container.resolve<OrderService>();
  service.process(Order(1, 'Заказ 1'));
}

Плюсы

  • Слабая связанность между классами
  • Простое тестирование (замена реализаций)
  • Гибкость и расширяемость
  • Централизованное управление зависимостями

Минусы

  • Усложняет код на ранних этапах (нужен composition root)
  • Может скрывать сложность графа зависимостей
  • Переусложнение для очень простых проектов

Связанные принципы

  • Dependency Inversion — DI это реализация DIP
  • Clean Architecture — DI необходим для соблюдения Dependency Rule
  • Singleton — DI-контейнеры часто управляют временем жизни объектов
