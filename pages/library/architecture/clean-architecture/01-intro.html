Clean Architecture (Чистая архитектура)

Также известен как Layered Architecture, Hexagonal Architecture
Автор Robert C. Martin (Uncle Bob)
Применимость ★★★★☆

Назначение

Создание архитектуры, независимой от фреймворков, баз данных и внешних библиотек. Бизнес-логика остаётся чистой и тестируемой, а детали реализации (UI, БД, API) становятся подключаемыми модулями.

Проблема

Традиционные приложения жёстко привязаны к фреймворку или базе данных. Смена БД или UI-фреймворка требует переписывания всего приложения. Тесты становятся медленными и хрупкими, так как требуют поднятия всей инфраструктуры. Бизнес-логика размазывается по слоям представления и данных.

Решение

Архитектура разделяется на концентрические слои с железным правилом зависимости: внутренний слой не знает о внешнем. В центре — сущности и бизнес-правила (use cases), снаружи — фреймворки, драйверы, UI.

  • Entities — бизнес-сущности (Enterprise Business Rules)
  • Use Cases — сценарии использования (Application Business Rules)
  • Interface Adapters — адаптеры (контроллеры, презентеры)
  • Frameworks & Drivers — внешние инструменты
  • Dependency Rule — зависимости направлены только внутрь

Пример на Dart

Структура проекта по Clean Architecture:

// ---- domain/entities ----
class User {
  final String id;
  final String name;
  final String email;
  User(this.id, this.name, this.email);
}

// ---- domain/usecases ----
abstract class UserRepository {
  Future<User> findById(String id);
  Future<void> save(User user);
}

class GetUserUseCase {
  final UserRepository repo;
  GetUserUseCase(this.repo);

  Future<User> execute(String id) => repo.findById(id);
}

// ---- data/repositories ----
class UserRepositoryImpl implements UserRepository {
  @override
  Future<User> findById(String id) async {
    // Работа с БД или API
    return User(id, 'Alice', 'alice@example.com');
  }

  @override
  Future<void> save(User user) async {
    print('Сохранён пользователь ${user.name}');
  }
}

// ---- presentation/controllers ----
class UserController {
  final GetUserUseCase getUser;

  UserController(this.getUser);

  Future<User?> showUser(String id) async {
    try {
      return await getUser.execute(id);
    } catch (e) {
      print('Ошибка: $e');
      return null;
    }
  }
}

Диаграмма зависимостей (слои):

// Frameworks (внешний слой)
//   ↓ depends on
// Interface Adapters
//   ↓ depends on
// Use Cases
//   ↓ depends on
// Entities (внутренний слой)

// Ни один внутренний слой не знает о внешних слоях!

Плюсы

  • Независимость от фреймворков и библиотек
  • Бизнес-логика тестируется без инфраструктуры
  • Возможность замены внешних компонентов (БД, UI)
  • Чёткое разделение ответственностей

Минусы

  • Больше кода и файлов для простых приложений
  • Требует дисциплины в соблюдении границ слоёв
  • Может быть избыточен для небольших проектов

Связанные принципы

  • Dependency Inversion — основа Dependency Rule
  • Dependency Injection — механизм реализации DIP
  • Domain-Driven Design — тесно связан с Clean Architecture
