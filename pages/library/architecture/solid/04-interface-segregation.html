SOLID

I. Interface Segregation (Принцип разделения интерфейсов)

Также известен как ISP
Сложность ★★☆☆☆
Применимость ★★★★☆

Назначение

Много специализированных интерфейсов лучше, чем один универсальный. Клиенты не должны зависеть от методов, которые они не используют.

Проблема

«Толстый» интерфейс (fat interface) содержит множество методов. Классы, реализующие его, вынуждены определять методы, которые им не нужны. Это приводит к пустым реализациям, выбросу исключений или возврату null, что нарушает LSP и увеличивает связанность.

Решение

Разделить «толстый» интерфейс на несколько узких, каждый для конкретной группы клиентов. Каждый класс реализует только те интерфейсы, которые ему действительно нужны.

Пример на Dart

Нарушение ISP — один «толстый» интерфейс:

abstract class Worker {
  void work();
  void eat();
  void sleep();
}

class Human implements Worker {
  @override void work() => print('Человек работает');
  @override void eat() => print('Человек ест');
  @override void sleep() => print('Человек спит');
}

// Робот вынужден реализовывать eat и sleep, которые ему не нужны
class Robot implements Worker {
  @override void work() => print('Робот работает');
  @override void eat() => throw UnimplementedError('Робот не ест');
  @override void sleep() => throw UnimplementedError('Робот не спит');
}

Решение — разделить на узкие интерфейсы:

abstract class Workable {
  void work();
}

abstract class Eatable {
  void eat();
}

abstract class Sleepable {
  void sleep();
}

class Human implements Workable, Eatable, Sleepable {
  @override void work() => print('Человек работает');
  @override void eat() => print('Человек ест');
  @override void sleep() => print('Человек спит');
}

// Робот реализует только то, что нужно
class Robot implements Workable {
  @override void work() => print('Робот работает');
}

Плюсы

  • Классы реализуют только то, что им действительно нужно
  • Снижается связанность между компонентами
  • Упрощается тестирование и мокирование
  • Повышается читаемость и понятность кода

Минусы

  • Увеличивается количество интерфейсов
  • Требуется продумывать границы интерфейсов заранее

Связанные принципы

  • Liskov Substitution — ISP помогает соблюдать LSP
  • Single Responsibility — ISP — это SRP для интерфейсов
