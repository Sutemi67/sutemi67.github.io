SOLID

D. Dependency Inversion (Принцип инверсии зависимостей)

Также известен как DIP
Сложность ★★★☆☆
Применимость ★★★★★

Назначение

Модули верхнего уровня не должны зависеть от модулей нижнего уровня. Оба должны зависеть от абстракций. Абстракции не должны зависеть от деталей — детали должны зависеть от абстракций.

Проблема

Когда высокоуровневый модуль (бизнес-логика) напрямую зависит от низкоуровневого модуля (база данных, API), изменение низкоуровневого модуля ломает бизнес-логику. Такой код сложно тестировать, переиспользовать и изменять.

Решение

Ввести абстракцию (интерфейс) между модулями. Высокоуровневый модуль работает через интерфейс, а низкоуровневый модуль реализует этот интерфейс. Направление зависимости «переворачивается»: оба модуля зависят от абстракции.

Пример на Dart

Нарушение DIP — бизнес-логика зависит от конкретной БД:

class PostgresDatabase {
  void save(String data) => print('Сохранение в PostgreSQL: $data');
}

class UserService {
  final PostgresDatabase db = PostgresDatabase(); // жёсткая связь

  void createUser(String name) {
    db.save('User: $name');
  }
}

После применения DIP — зависимость от абстракции:

abstract class Database {
  void save(String data);
}

class PostgresDatabase implements Database {
  @override
  void save(String data) => print('Сохранение в PostgreSQL: $data');
}

class MongoDatabase implements Database {
  @override
  void save(String data) => print('Сохранение в MongoDB: $data');
}

class UserService {
  final Database db; // зависит от абстракции

  UserService(this.db); // внедрение зависимости

  void createUser(String name) {
    db.save('User: $name');
  }
}

void main() {
  final service = UserService(PostgresDatabase());
  service.createUser('Alice');

  // Легко заменить реализацию:
  final testService = UserService(MongoDatabase());
  testService.createUser('Bob');
}

Плюсы

  • Снижается связанность между модулями
  • Упрощается тестирование (легко подменять реализации)
  • Возможность замены реализаций без изменения клиентского кода
  • Код становится гибким и расширяемым

Минусы

  • Увеличивается сложность из-за дополнительных абстракций
  • Требует использования DI-контейнеров или фабрик

Связанные принципы

  • Dependency Injection — практическая реализация DIP
  • Open-Closed — DIP является основой для OCP
  • Strategy — паттерн, реализующий DIP на практике
