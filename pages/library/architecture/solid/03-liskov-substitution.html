SOLID

L. Liskov Substitution (Принцип подстановки Барбары Лисков)

Также известен как LSP
Сложность ★★★☆☆
Применимость ★★★★☆

Назначение

Объекты базового класса должны быть заменяемы объектами его подклассов без нарушения корректности программы. Если код работает с базовым типом, он должен работать и с любым его подтипом, не зная об этом.

Проблема

Классический пример — иерархия Rectangle → Square. Квадрат является частным случаем прямоугольника, но изменение ширины квадрата меняет и высоту, что нарушает поведение, ожидаемое от прямоугольника. Клиентский код, работающий с Rectangle, сломается при подстановке Square.

Решение

Не наследовать там, где нарушается контракт базового класса. Вместо наследования использовать композицию или общий абстрактный интерфейс. Контракт (предусловия и постусловия) базового класса должен соблюдаться всеми подклассами.

Пример на Dart

Нарушение LSP:

class Rectangle {
  double width, height;
  Rectangle(this.width, this.height);
}

class Square extends Rectangle {
  Square(double side) : super(side, side);

  @override
  set width(double value) {
    super.width = value;
    super.height = value; // нарушение: меняем и высоту
  }

  @override
  set height(double value) {
    super.height = value;
    super.width = value; // нарушение: меняем и ширину
  }
}

void processRectangle(Rectangle rect) {
  rect.width = 5;
  rect.height = 10;
  assert(rect.area() == 50); // для Square упадёт!
}

Решение — общий абстрактный интерфейс вместо наследования:

abstract class Shape {
  double area();
}

class Rectangle implements Shape {
  double width, height;
  Rectangle(this.width, this.height);

  @override
  double area() => width * height;
}

class Square implements Shape {
  double side;
  Square(this.side);

  @override
  double area() => side * side;
}

void processShape(Shape shape) {
  print('Площадь: ${shape.area()}');
}

Плюсы

  • Гарантирует корректное поведение полиморфного кода
  • Упрощает замену реализаций без побочных эффектов
  • Повышает надёжность и предсказуемость кода

Минусы

  • Требует тщательного проектирования иерархий наследования
  • Не всегда очевидно на ранних этапах разработки

Связанные принципы

  • Open-Closed — LSP необходим для корректного расширения через OCP
  • Interface Segregation — узкие интерфейсы упрощают соблюдение LSP
