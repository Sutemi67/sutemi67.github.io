SOLID

O. Open-Closed (Принцип открытости/закрытости)

Также известен как OCP
Сложность ★★☆☆☆
Применимость ★★★★★

Назначение

Классы должны быть открыты для расширения, но закрыты для изменения. Новую функциональность следует добавлять через новый код, не изменяя существующий.

Проблема

Представьте модуль расчёта площади фигур. При добавлении новой фигуры (например, треугольника) приходится изменять существующий код — добавлять новую ветку if или switch. Каждое такое изменение рискует сломать уже работающую логику.

Решение

Использовать абстракции (интерфейсы или абстрактные классы). Базовый код работает через общий интерфейс, а конкретная реализация добавляется новым классом, не затрагивая существующий.

Пример на Dart

Нарушение OCP — при добавлении новой фигуры нужно менять AreaCalculator:

class AreaCalculator {
  double calculate(Object shape) {
    if (shape is Circle) {
      return 3.14 * shape.radius * shape.radius;
    } else if (shape is Rectangle) {
      return shape.width * shape.height;
    }
    throw ArgumentError('Неизвестная фигура');
  }
}

После применения OCP — расширение через новые классы:

abstract class Shape {
  double area();
}

class Circle implements Shape {
  final double radius;
  Circle(this.radius);

  @override
  double area() => 3.14159 * radius * radius;
}

class Rectangle implements Shape {
  final double width, height;
  Rectangle(this.width, this.height);

  @override
  double area() => width * height;
}

// Новая фигура — без изменения существующего кода
class Triangle implements Shape {
  final double base, height;
  Triangle(this.base, this.height);

  @override
  double area() => 0.5 * base * height;
}

class AreaCalculator {
  double calculate(Shape shape) => shape.area();
}

Плюсы

  • Существующий код не изменяется — меньше риска регрессий
  • Новая функциональность добавляется изолированно
  • Упрощается тестирование (каждый класс тестируется отдельно)

Минусы

  • Требует проектирования абстракций заранее
  • Может привести к избыточному количеству классов

Связанные принципы

  • Liskov Substitution — реализации должны быть взаимозаменяемы
  • Dependency Inversion — OCP тесно связан с инверсией зависимостей
  • Strategy — паттерн, реализующий OCP на практике
