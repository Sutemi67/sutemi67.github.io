SOLID

S. Single Responsibility (Принцип единственной ответственности)

Также известен как SRP
Сложность ★☆☆☆☆
Применимость ★★★★★

Назначение

Каждый класс должен иметь только одну причину для изменения. Это означает, что класс должен отвечать только за одну строго определённую часть функциональности программы.

Проблема

Когда класс берёт на себя несколько обязанностей, изменение одной из них может привести к поломке других. Такой класс сложно тестировать, поддерживать и расширять. Например, класс OrderProcessor, который и вычисляет итоговую сумму, и сохраняет заказ в базу данных, и отправляет email — нарушает SRP.

Решение

Разделить класс на несколько, каждый со своей зоной ответственности. Выделить в отдельные классы: бизнес-логику (расчёт суммы), работу с хранилищем (сохранение), отправку уведомлений (email). Каждый такой класс будет иметь только одну причину для изменения.

Пример на Dart

Нарушение SRP — всё в одном классе:

class OrderProcessor {
  double calculateTotal(Order order) {
    return order.items.fold(0, (sum, item) => sum + item.price);
  }

  void saveToDatabase(Order order) {
    print('Сохранение заказа ${order.id} в БД');
  }

  void sendEmailNotification(Order order) {
    print('Отправка email о заказе ${order.id}');
  }
}

После применения SRP — каждая ответственность в своём классе:

class OrderCalculator {
  double calculateTotal(Order order) {
    return order.items.fold(0, (sum, item) => sum + item.price);
  }
}

class OrderRepository {
  void save(Order order) {
    print('Сохранение заказа ${order.id} в БД');
  }
}

class EmailService {
  void sendOrderConfirmation(Order order) {
    print('Отправка email о заказе ${order.id}');
  }
}

Плюсы

  • Код проще поддерживать и изменять
  • Каждый класс легче тестировать изолированно
  • Снижается связанность (coupling) между компонентами
  • Упрощается понимание кода новыми разработчиками

Минусы

  • Увеличивается количество классов в проекте
  • Требуется более тщательное проектирование архитектуры

Связанные принципы

  • Open-Closed — SRP помогает достичь OCP, так как классы с одной ответственностью проще расширять
  • Dependency Injection — разделение ответственностей упрощает внедрение зависимостей
