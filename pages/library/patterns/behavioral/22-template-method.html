Template Method (Шаблонный метод)

Также известен как Template Method
Сложность ★☆☆☆☆
Популярность ★★★☆☆

Назначение

Определяет скелет алгоритма в базовом классе, позволяя подклассам переопределять отдельные шаги без изменения структуры алгоритма.

Проблема

Нужно реализовать парсеры данных разных форматов (CSV, JSON, XML). Алгоритм парсинга одинаков: открыть файл, прочитать данные, разобрать, обработать ошибки, закрыть файл. Отличается только этап разбора. Если копировать весь алгоритм для каждого формата, код будет дублироваться.

Решение

Базовый класс определяет шаблонный метод с последовательностью шагов. Некоторые шаги — абстрактные (должны быть реализованы в подклассах), другие — имеют реализацию по умолчанию. Подклассы переопределяют только необходимые шаги.

Пример на Dart

abstract class DataParser {
  void parse(String filepath) {
    openFile(filepath);
    final rawData = readFile();
    final parsedData = parseData(rawData);
    processData(parsedData);
    closeFile();
  }

  void openFile(String path) {
    print('Открытие файла: $path');
  }

  String readFile() {
    print('Чтение данных из файла');
    return 'raw data';
  }

  void closeFile() {
    print('Закрытие файла');
  }

  List> parseData(String data);

  void processData(List> data) {
    print('Обработано ${data.length} записей');
  }
}

class CsvParser extends DataParser {
  @override
  List> parseData(String data) {
    print('Разбор CSV: разделение по запятым');
    return [
      {'name': 'Alice', 'age': 30},
      {'name': 'Bob', 'age': 25},
    ];
  }
}

class JsonParser extends DataParser {
  @override
  List> parseData(String data) {
    print('Разбор JSON: декодирование строки');
    return [
      {'name': 'Charlie', 'age': 35},
    ];
  }

  @override
  void processData(List> data) {
    print('JSON-валидация пройдена');
    super.processData(data);
  }
}

void main() {
  final csvParser = CsvParser();
  csvParser.parse('data.csv');

  print('---');

  final jsonParser = JsonParser();
  jsonParser.parse('data.json');
}

Когда применять

  • Когда несколько классов имеют одинаковый алгоритм, но разную реализацию отдельных шагов
  • Когда нужно избежать дублирования кода в подклассах
  • Когда нужно контролировать, какие шаги обязательны, а какие опциональны

Плюсы

  • Избегает дублирования кода (DRY)
  • Позволяет подклассам переопределять только нужные шаги
  • Контролирует структуру алгоритма из базового класса

Минусы

  • Ограничивает гибкость — подклассы следуют жёсткому скелету
  • Может нарушать принцип подстановки Лисков (LSP)

Связанные паттерны

  • Strategy — заменяет целые алгоритмы; Template Method задаёт скелет с переопределяемыми шагами
  • Factory Method — частный случай Template Method
