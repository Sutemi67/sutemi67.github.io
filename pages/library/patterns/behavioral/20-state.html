State (Состояние)

Также известен как Finite State Machine
Сложность ★★★☆☆
Популярность ★★★☆☆

Назначение

Позволяет объекту изменять своё поведение в зависимости от внутреннего состояния, создавая иллюзию смены класса.

Проблема

Заказ в интернет-магазине проходит через несколько состояний: Новый → Оплачен → Отправлен → Доставлен. В каждом состоянии доступны разные действия. Реализация через условные операторы (if/switch) приводит к раздуванию кода и сложности поддержки.

Решение

Каждое состояние выносится в отдельный класс. Контекст (заказ) делегирует выполнение действий текущему объекту-состоянию. Смена состояния — это подмена объекта состояния на другой.

Пример на Dart

abstract class OrderState {
  void pay(OrderContext context);
  void ship(OrderContext context);
  void deliver(OrderContext context);
  String get name;
}

class NewOrderState implements OrderState {
  @override
  String get name => 'Новый';

  @override
  void pay(OrderContext context) {
    print('Заказ оплачен');
    context.setState(PaidOrderState());
  }

  @override
  void ship(OrderContext context) {
    print('Нельзя отправить — заказ не оплачен');
  }

  @override
  void deliver(OrderContext context) {
    print('Нельзя доставить — заказ не оплачен');
  }
}

class PaidOrderState implements OrderState {
  @override
  String get name => 'Оплачен';

  @override
  void pay(OrderContext context) => print('Заказ уже оплачен');

  @override
  void ship(OrderContext context) {
    print('Заказ отправлен');
    context.setState(ShippedOrderState());
  }

  @override
  void deliver(OrderContext context) => print('Заказ ещё не отправлен');
}

class ShippedOrderState implements OrderState {
  @override
  String get name => 'Отправлен';

  @override
  void pay(OrderContext context) => print('Заказ уже оплачен');

  @override
  void ship(OrderContext context) => print('Заказ уже отправлен');

  @override
  void deliver(OrderContext context) {
    print('Заказ доставлен');
    context.setState(DeliveredOrderState());
  }
}

class DeliveredOrderState implements OrderState {
  @override
  String get name => 'Доставлен';

  @override
  void pay(OrderContext context) => print('Заказ уже оплачен');
  @override
  void ship(OrderContext context) => print('Заказ уже доставлен');
  @override
  void deliver(OrderContext context) => print('Заказ уже доставлен');
}

class OrderContext {
  OrderState _state = NewOrderState();

  void setState(OrderState state) => _state = state;
  String get status => _state.name;

  void pay() => _state.pay(this);
  void ship() => _state.ship(this);
  void deliver() => _state.deliver(this);
}

void main() {
  final order = OrderContext();
  order.ship();
  order.pay();
  order.ship();
  order.deliver();
  print('Статус: ${order.status}');
}

Когда применять

  • Когда поведение объекта зависит от его состояния и меняется во время выполнения
  • Когда есть множество условных операторов, проверяющих состояние
  • Когда состояния имеют чёткие границы и разные правила переходов

Плюсы

  • Избавляет от громоздких условных конструкций
  • Каждое состояние — отдельный класс (SRP)
  • Упрощает добавление новых состояний

Минусы

  • Может привести к большому количеству классов
  • Усложняет код для простых конечных автоматов

Связанные паттерны

  • Strategy — похожая структура, но разная цель: Strategy для взаимозаменяемых алгоритмов, State для управления состоянием
  • Observer — может уведомлять об изменении состояния
