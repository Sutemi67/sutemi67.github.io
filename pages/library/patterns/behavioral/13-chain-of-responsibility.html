Chain of Responsibility (Цепочка обязанностей)

Также известен как CoR, Middleware
Сложность ★★☆☆☆
Популярность ★★★★☆

Назначение

Позволяет передавать запросы последовательно по цепочке обработчиков, пока один из них не обработает запрос.

Проблема

Веб-сервер должен пропускать HTTP-запрос через цепочку middleware: логирование, аутентификация, сжатие, кеширование, роутинг. Каждый middleware может либо обработать запрос, либо передать его дальше. Добавление новых middleware не должно требовать изменения кода существующих.

Решение

Каждый обработчик реализует общий интерфейс и имеет ссылку на следующий обработчик в цепочке. Обработчик решает, обработать запрос или передать следующему. Цепочка строится динамически.

Пример на Dart

abstract class Middleware {
  Middleware? _next;

  void linkNext(Middleware next) => _next = next;

  void handle(String request);

  void next(String request) {
    _next?.handle(request);
  }
}

class LoggingMiddleware extends Middleware {
  @override
  void handle(String request) {
    print('[LOG] Запрос: $request');
    next(request);
  }
}

class AuthMiddleware extends Middleware {
  @override
  void handle(String request) {
    if (request.contains('token')) {
      print('[AUTH] Доступ разрешён');
      next(request);
    } else {
      print('[AUTH] Доступ запрещён');
    }
  }
}

class CacheMiddleware extends Middleware {
  final Set _cache = {};

  @override
  void handle(String request) {
    if (_cache.contains(request)) {
      print('[CACHE] Ответ из кеша');
    } else {
      _cache.add(request);
      next(request);
    }
  }
}

void main() {
  final logging = LoggingMiddleware();
  final auth = AuthMiddleware();
  final cache = CacheMiddleware();

  logging.linkNext(auth);
  auth.linkNext(cache);

  logging.handle('GET /api/data?token=123');
}

Когда применять

  • Когда есть несколько обработчиков, которые могут обработать запрос
  • Когда порядок обработки важен и может меняться
  • Когда набор обработчиков должен формироваться динамически

Плюсы

  • Уменьшает связанность отправителя и получателя
  • Гибкое добавление/изменение обработчиков
  • Реализует принцип единственной ответственности

Минусы

  • Нет гарантии, что запрос будет обработан
  • Сложно отслеживать прохождение запроса по цепочке

Связанные паттерны

  • Command — может использоваться для инкапсуляции запроса
  • Mediator — управляет коммуникацией, а не передаёт по цепочке
