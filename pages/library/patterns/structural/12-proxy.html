Proxy (Заместитель)

Также известен как Surrogate
Сложность ★★☆☆☆
Популярность ★★★★☆

Назначение

Предоставляет объект-заместитель, который контролирует доступ к другому объекту (реальному субъекту), позволяя перехватывать вызовы и добавлять дополнительную логику.

Проблема

Приложение отображает изображения в документе. Загрузка всех изображений сразу при открытии документа делает его запуск слишком медленным. Нужно отложить загрузку до момента, когда пользователь действительно увидит изображение.

Решение

Создаётся прокси-объект с тем же интерфейсом, что и реальный объект. Прокси перехватывает вызовы и выполняет дополнительную логику (ленивая загрузка, проверка доступа, кеширование) до или после делегирования реальному объекту.

Пример на Dart

abstract class Image {
  void display();
}

class RealImage implements Image {
  final String filename;

  RealImage(this.filename) {
    _loadFromDisk();
  }

  void _loadFromDisk() {
    print('Загрузка $filename с диска...');
  }

  @override
  void display() {
    print('Отображение $filename');
  }
}

class ProxyImage implements Image {
  final String filename;
  RealImage? _realImage;

  ProxyImage(this.filename);

  @override
  void display() {
    _realImage ??= RealImage(filename);
    _realImage!.display();
  }
}

class ProtectedImage implements Image {
  final RealImage _image;
  final bool _hasAccess;

  ProtectedImage(this._image, this._hasAccess);

  @override
  void display() {
    if (_hasAccess) {
      _image.display();
    } else {
      print('Доступ к изображению запрещён');
    }
  }
}

void main() {
  final image = ProxyImage('photo.jpg');
  print('Изображение создано, но не загружено');
  image.display();
  image.display();
}

Когда применять

  • Ленивая инициализация (виртуальный прокси) — создание объекта по требованию
  • Защита доступа (защитный прокси) — контроль прав доступа
  • Логирование (логирующий прокси) — запись вызовов методов
  • Кеширование (кеширующий прокси) — повторное использование результатов

Плюсы

  • Контроль доступа к объекту без изменения его кода
  • Возможность добавлять логику до/после вызова
  • Работает без изменения клиентского кода

Минусы

  • Добавляет прослойку, что может незначительно снизить производительность
  • Усложняет код из-за дополнительных классов

Связанные паттерны

  • Adapter — предоставляет другой интерфейс; Proxy — тот же интерфейс
  • Decorator — похож по структуре, но добавляет поведение, а не контролирует доступ
