Abstract Factory (Абстрактная фабрика)

Также известен как Kit
Сложность ★★★☆☆
Популярность ★★★★☆

Назначение

Предоставляет интерфейс для создания семейств взаимосвязанных или взаимозависимых объектов, не специфицируя их конкретные классы.

Проблема

Приложение должно работать с разными наборами UI-компонентов: кнопками, текстовыми полями, чекбоксами. Для каждой платформы (Material Design, Cupertino) нужен свой набор виджетов, но клиентский код не должен зависеть от конкретных реализаций.

Решение

Создаётся абстрактная фабрика с методами для создания каждого типа компонента. Для каждой платформы реализуется своя конкретная фабрика. Клиентский код получает фабрику и через неё создаёт компоненты, не зная их конкретных классов.

Пример на Dart

abstract class Button {
  void paint();
}

abstract class TextField {
  void render();
}

class MaterialButton implements Button {
  @override
  void paint() => print('Material Button');
}

class CupertinoButton implements Button {
  @override
  void paint() => print('Cupertino Button');
}

class MaterialTextField implements TextField {
  @override
  void render() => print('Material TextField');
}

class CupertinoTextField implements TextField {
  @override
  void render() => print('Cupertino TextField');
}

abstract class UIFactory {
  Button createButton();
  TextField createTextField();
}

class MaterialFactory implements UIFactory {
  @override
  Button createButton() => MaterialButton();
  @override
  TextField createTextField() => MaterialTextField();
}

class CupertinoFactory implements UIFactory {
  @override
  Button createButton() => CupertinoButton();
  @override
  TextField createTextField() => CupertinoTextField();
}

void main() {
  UIFactory factory = MaterialFactory();
  final button = factory.createButton();
  final textField = factory.createTextField();
  button.paint();
  textField.render();
}

Когда применять

  • Когда система не должна зависеть от способа создания компонентов
  • Когда нужно создавать семейства взаимосвязанных объектов
  • Когда есть несколько семейств объектов, которые могут меняться местами

Плюсы

  • Гарантирует совместимость создаваемых объектов
  • Избавляет от жёсткой привязки к конкретным классам
  • Упрощает замену целых семейств продуктов

Минусы

  • Сложно добавить новый тип продукта (нужно менять все фабрики)
  • Увеличивает количество классов и интерфейсов

Связанные паттерны

  • Factory Method — методы фабрики часто реализуются как Factory Method
  • Singleton — конкретная фабрика часто делается синглтоном
