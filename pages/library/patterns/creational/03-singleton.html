Singleton (Одиночка)

Также известен как Одиночка
Сложность ★☆☆☆☆
Популярность ★★★★★

Назначение

Гарантирует, что класс имеет только один экземпляр, и предоставляет глобальную точку доступа к этому экземпляру.

Проблема

В приложении нужен единый объект для управления конфигурацией, логированием или подключением к базе данных. Если создать несколько экземпляров, это приведёт к конфликтам, избыточному использованию ресурсов или несогласованному состоянию.

Решение

Класс сам контролирует создание своего единственного экземпляра. Конструктор делается приватным, а доступ к экземпляру — через статический метод. В Dart это можно реализовать через factory-конструктор или статическое поле.

Пример на Dart

class AppConfig {
  static final AppConfig _instance = AppConfig._internal();

  factory AppConfig() => _instance;

  AppConfig._internal();

  String apiUrl = 'https://api.example.com';
  int timeout = 30;
  bool debugMode = false;

  void load() {
    print('Загрузка конфигурации...');
  }
}

class Logger {
  static Logger? _instance;

  Logger._();

  static Logger get instance {
    _instance ??= Logger._();
    return _instance!;
  }

  void log(String message) {
    print('[LOG] $message');
  }
}

class DatabaseConnection {
  static DatabaseConnection? _instance;
  bool _connected = false;

  DatabaseConnection._();

  static DatabaseConnection getInstance() {
    _instance ??= DatabaseConnection._();
    return _instance!;
  }

  void connect() {
    if (!_connected) {
      print('Подключение к БД...');
      _connected = true;
    }
  }
}

void main() {
  final config1 = AppConfig();
  final config2 = AppConfig();
  print(identical(config1, config2)); // true

  Logger.instance.log('Приложение запущено');
  DatabaseConnection.getInstance().connect();
}

Когда применять

  • Когда нужен ровно один экземпляр класса (логгер, конфиг, пул соединений)
  • Когда требуется строгий контроль над доступом к общему ресурсу
  • Когда создание объекта дорого и его нужно отложить до первого использования

Плюсы

  • Гарантирует единственность экземпляра
  • Ленивая инициализация (создание при первом обращении)
  • Глобальная точка доступа

Минусы

  • Нарушает принцип единственной ответственности
  • Создаёт глобальное состояние, усложняя тестирование
  • Может привести к скрытым зависимостям

Связанные паттерны

  • Factory Method — фабрика может быть реализована как синглтон
  • Proxy — может управлять доступом к объекту-одиночке
