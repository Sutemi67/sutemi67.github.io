Factory Method (Фабричный метод)

Также известен как Virtual Constructor
Сложность ★★☆☆☆
Популярность ★★★★★

Назначение

Определяет интерфейс для создания объекта, но позволяет подклассам изменять тип создаваемых объектов.

Проблема

Представьте, что вы пишете приложение для управления логистикой. Сначала оно работало только с наземными перевозками (грузовики), но затем появляется потребность добавить морские и воздушные перевозки. Код создания всех видов транспорта размазан по всему приложению, и каждое добавление нового типа требует изменения существующего кода.

Решение

Фабричный метод предлагает вынести код создания объектов в отдельный метод, объявленный в базовом классе (или интерфейсе). Подклассы могут переопределять этот метод, создавая объекты конкретных типов. Клиентский код работает через общий интерфейс и не зависит от конкретных классов.

Пример на Dart

abstract class Button {
  void render();
  void onClick(void Function() callback);
}

class WindowsButton implements Button {
  @override
  void render() => print('Рисуем кнопку в стиле Windows');

  @override
  void onClick(void Function() callback) {
    print('Назначен обработчик для Windows');
  }
}

class LinuxButton implements Button {
  @override
  void render() => print('Рисуем кнопку в стиле Linux');

  @override
  void onClick(void Function() callback) {
    print('Назначен обработчик для Linux');
  }
}

abstract class Dialog {
  Button createButton();

  void render() {
    final button = createButton();
    button.render();
    button.onClick(() => print('Клик!'));
  }
}

class WindowsDialog extends Dialog {
  @override
  Button createButton() => WindowsButton();
}

class LinuxDialog extends Dialog {
  @override
  Button createButton() => LinuxButton();
}

void main() {
  Dialog dialog = WindowsDialog();
  dialog.render();

  dialog = LinuxDialog();
  dialog.render();
}

Когда применять

  • Когда заранее неизвестно, экземпляры каких классов нужно создавать
  • Когда класс хочет, чтобы его подклассы определяли создаваемые объекты
  • Когда нужно локализовать код создания объектов в одном месте

Плюсы

  • Избавляет от жёсткой привязки к конкретным классам
  • Код создания объектов собирается в одном месте
  • Упрощает добавление новых типов продуктов
  • Реализует принцип открытости/закрытости

Минусы

  • Может привести к появлению множества подклассов
  • Усложняет код из-за дополнительных интерфейсов

Связанные паттерны

  • Abstract Factory — часто реализуется через Factory Method
  • Singleton — можно сделать фабрику синглтоном
